# Development Document Rules

このドキュメントは、IroMap の SDD（Specification Driven Development）で使用する
`spec.md` / `design.md` / `tasks.md` / `ADR` の標準フォーマットを定義する。

## 1. 基本原則
- README は概要のみを扱い、機能詳細は `docs/specs/` 配下に記載する
- 1 機能 = 1 ディレクトリ（`docs/specs/YYYY-MM-feature-name/`）
- 要求（What）と設計（How）を分離する
- 各タスクは要求 ID と紐づける
- 未決事項を隠さず `未決事項` または `notes.md` に残す

## 2. spec.md フォーマット（要求定義）

```md
# Feature Spec: <機能名>

## 1. 背景
- 何の課題を解決したいか
- なぜ今やるのか
- やらないと何が困るか

## 2. スコープ
### In Scope
- ...

### Out of Scope
- ...

## 3. 利用者 / ステークホルダー
- ...

## 4. ユーザーストーリー
- ...

## 5. 機能要求
### FR-1
...

#### 受け入れ条件
- ...

### FR-2
...

#### 受け入れ条件
- ...

## 6. 非機能要求
- セキュリティ: ...
- 性能: ...
- 可観測性: ...

## 7. 業務ルール / 制約
- ...

## 8. エッジケース
- ...

## 9. 成功指標
- ...

## 10. 未決事項
- ...
```

### 書き方の要点
- 背景: 「なぜ」
- スコープ: 「どこまで」
- 機能要求: 「何をする」
- 受け入れ条件: 「どうなれば OK か」
- 非機能要求: 「どの品質を守るか」
- 未決事項: 「まだ決まっていないこと」

## 3. design.md フォーマット（実現方式）

```md
# Design: <機能名>

## 1. 目的
この設計は spec.md の FR-x を満たすための実装方針を定義する。

## 2. 全体方針
- ...

## 3. アーキテクチャ上の位置づけ
- ...

## 4. ドメインモデル
### EntityA
- ...

## 5. シーケンス
1. ...

## 6. API設計
### POST /...
request:
- ...

response:
- ...

## 7. 例外系
- ...

## 8. セキュリティ考慮
- ...

## 9. 可観測性
- ...

## 10. トレードオフ
- ...

## 11. 関連ADR
- ADR-xxxx ...
```

### 設計で必ず書くこと
- 要求を満たす具体手段
- 境界条件と例外時の扱い
- 採用案の利点と欠点（トレードオフ）

## 4. tasks.md フォーマット（実装計画）

```md
# Tasks: <機能名>

## ルール
- 各タスクは spec の要求IDに紐づける
- 各タスクは完了条件を持つ
- 実装、テスト、監視を分ける

## 実装
- [ ] T-001 ...（FR-1）
  - 完了条件: ...

## テスト
- [ ] T-101 ...（FR-1）

## 監視 / 運用
- [ ] T-201 ...（NFR-observability）
```

### タスク記述ルール
- ID は `T-xxx` で一意
- 要求トレーサビリティを失わない
- 完了条件は検証可能な文で書く

## 5. ADR フォーマット（技術判断）

```md
# ADR-xxxx: <決定タイトル>

## Status
Accepted | Proposed | Deprecated

## Context
...

## Decision
...

## Consequences
### Positive
- ...

### Negative
- ...

## Alternatives Considered
- ...
```

### ADR 運用ルール
- 1 ADR = 1 決定
- 短く保つ（背景と判断理由を最小十分で記録）
- 関連機能の `design.md` から参照する

## 6. 品質観点
- 要求は曖昧語を避け、検証可能に書く
- 性能値・期限・回数制限は数値で記載する
- `spec -> design -> tasks -> test-cases` の整合を維持する
