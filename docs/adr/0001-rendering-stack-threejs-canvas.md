# ADR-0001: 描画スタックとして Three.js + Canvas を採用する

## Status
Accepted

## Context
- IroMap は RGB 色空間を 3D で把握しつつ、任意断面を 2D で確認することを主目的とする。
- 3D は「グラフ」ではなく「空間表現」が必要であり、自由なカメラ操作と表現方式の拡張余地が求められる。
- 2D Slice は 1px 単位の色制御と高速な再描画が必要であり、チャート用途ライブラリでは自由度が不足しやすい。
- `docs/architecture/quality-attributes.md` の性能要件（体感遅延の最小化）と拡張性要件（将来の色空間追加）を両立する必要がある。

## Decision
- 3D 描画ライブラリに Three.js を採用する。
- 2D Slice 描画に HTML Canvas（`ImageData` ベース）を採用する。
- RGB キューブ表現の初期実装は Three.js の `InstancedMesh` を基本方針とする。

## Consequences
### Positive
- RGB 空間を 3D オブジェクトとして表現しやすく、回転・断面同期・将来拡張の自由度が高い。
- 2D Slice はピクセル単位で制御でき、補間や配色ロジックを実装側で明示できる。
- 3D と 2D の責務分離が明確になり、仕様変更時の影響範囲を切り分けやすい。

### Negative
- Three.js のシーン管理、メモリ管理、React 連携の実装コストが発生する。
- Canvas は宣言的 UI と異なり、描画更新の責務を明示的に管理する必要がある。
- 3D と 2D で描画基盤が分かれるため、座標系と同期仕様を設計で固定しないと不整合リスクがある。

## Performance Criteria
- 3D キューブの初回表示は操作待ちが発生しない時間で完了すること（目安: 1 秒以内）。
- 3D 回転操作は連続操作時に視認可能な追従性を維持すること（目安: 30 FPS 以上）。
- Slice 固定値変更時の 2D 再描画は操作遅延を感じにくい時間で完了すること（目安: 50 ms 以内）。
- 上記の具体的な測定条件・閾値・検証手順は各 `spec.md` / `test-cases.md` で管理する。

## Alternatives Considered
- deck.gl:
  - 長所: WebGL ベースで大規模データ描画に強い。
  - 非採用理由: RGB 空間オブジェクトの細かな制御と表現自由度は Three.js の方が高い。
- VTK.js:
  - 長所: 科学可視化の機能が豊富。
  - 非採用理由: 本プロジェクトの MVP 範囲では機能過多で、導入/運用コストが相対的に高い。
- チャート系 3D ライブラリ:
  - 非採用理由: RGB 空間をグラフとして扱う前提が強く、空間表現との整合が低い。
- 2D を WebGL 系（regl など）で統一:
  - 長所: GPU 活用の余地がある。
  - 非採用理由: 現段階では Canvas + ImageData で要求を満たせる見込みが高く、実装複雑性を抑えられる。
