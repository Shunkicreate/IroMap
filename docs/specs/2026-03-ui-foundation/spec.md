# Feature Spec: UI Foundation (shadcn/ui + Radix)

## 1. 背景
- 既存の初期テンプレートUIを、分析ワークベンチ向けの再利用可能なUI基盤へ置き換える必要がある
- 今後の機能実装で一貫したアクセシビリティとデザイントークン運用を維持したい

## 2. スコープ
### In Scope
- shadcn/ui + Radix を導入し、共通UIコンポーネント基盤を整備する
- トップページを IroMap 向け1ページ構成に置き換える
- ライト/ダークテーマ切替を提供する

### Out of Scope
- RGB Cube / Slice / Photo Analysis の本体実装
- 複数ルートへの画面分割
- デザインシステムの完全固定（ブランド最終調整）

## 3. 利用者 / ステークホルダー
- 写真分析機能を利用するエンドユーザー
- 今後UIを実装する開発者

## 4. ユーザーストーリー
- ユーザーとして、分析UIの全体像を1ページで把握したい。なぜなら機能の理解を素早く行いたいから
- 開発者として、共通コンポーネントを使って機能実装したい。なぜならUI品質と実装速度を両立したいから

## 5. 機能要求
### FR-1: shadcn/ui + Radix のUI基盤を利用できる
#### 受け入れ条件
- Button/Card/Input/Select/Tabs/Separator/Skeleton/Tooltip/Toast を利用できる
- デザイントークンが `globals.css` で定義されている

### FR-2: トップページを IroMap 向け構成へ置き換える
#### 受け入れ条件
- Hero / Workbench Preview / Feature Cards / Docs 導線が1ページ内に配置される
- 既存の create-next-app 初期デザインが表示されない

### FR-3: ライト/ダークテーマを切り替えできる
#### 受け入れ条件
- UI操作でテーマが切り替えられる
- テーマ切替後も主要コンポーネントの可読性が維持される

## 6. 非機能要求
- NFR-1: キーボード操作で Select/Tabs/Tooltip の基本操作が可能である
- NFR-2: モバイル幅（390px想定）で主要情報が読める

## 7. 業務ルール / 制約
- 依存方向 `app -> features -> domain/rendering/lib` の方針を維持する
- 本機能では分析ロジックを追加しない

## 8. エッジケース
- テーマ切替直後に hydration mismatch を発生させない
- プレースホルダUIでもレイアウト崩れを起こさない

## 9. 成功指標
- 今後の画面実装で共通UIを再利用できる状態になる
- 1ページ上で機能ロードマップを説明できる

## 10. 未決事項
- なし
