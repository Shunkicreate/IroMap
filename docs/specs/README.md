# Specs README

`docs/specs/` は機能単位の仕様セットを管理するフォルダです。

## 意図
- 1 機能 = 1 ディレクトリで要求から実装計画まで追跡可能にする
- `spec -> design -> tasks -> test-cases` の整合を維持する

## ディレクトリ命名
- `YYYY-MM-<feature-name>`
- 例: `2026-03-login`

## 各機能ディレクトリの標準構成
- `spec.md`: 要求定義（What）
- `design.md`: 実現方式（How）
- `tasks.md`: 実装計画（Work items）
- `test-cases.md`: 受け入れ観点
- `notes.md`: 未決事項・補足メモ

## 推奨フォーマット
以下は `2026-03-login` / `2026-03-user-registration` で使っていた書き方を統合したテンプレート。

### `spec.md`
```md
# Feature Spec: <機能名>

## 1. 背景
## 2. スコープ
### In Scope
### Out of Scope
## 3. 利用者 / ステークホルダー
## 4. ユーザーストーリー
## 5. 機能要求
### FR-1
#### 受け入れ条件
## 6. 非機能要求
## 7. 業務ルール / 制約
## 8. エッジケース
## 9. 成功指標
## 10. 未決事項
```

### `design.md`
```md
# Design: <機能名>

## 1. 目的
## 2. 全体方針
## 3. アーキテクチャ上の位置づけ
## 4. ドメインモデル
## 5. シーケンス
## 6. API設計
## 7. 例外系
## 8. セキュリティ考慮
## 9. 可観測性
## 10. トレードオフ
## 11. 関連ADR
```

### `tasks.md`
```md
# Tasks: <機能名>

## ルール
- 各タスクは spec の要求 ID に紐づける
- 各タスクは完了条件を持つ
- 実装、テスト、監視を分けて書く

## 実装
- [ ] T-001 ...（FR-1）
  - 完了条件: ...

## テスト
- [ ] T-101 ...（FR-1）

## 監視 / 運用
- [ ] T-201 ...（NFR-observability）
```

### `test-cases.md`
```md
# Test Cases: <機能名>

## 対応要求
- FR-1

## 正常系
- TC-001: ...（FR-1）

## 異常系
- TC-101: ...（FR-1）
```

### `notes.md`
```md
# Notes: <機能名>

- 未決事項・補足メモ
```

## 書き方の意味
- `spec.md` は「何を満たすか」を定義する
- `design.md` は「どう満たすか」を定義する
- `tasks.md` と `test-cases.md` は要求 ID に紐づける
