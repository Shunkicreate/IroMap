# Feature Spec: AI Agent対応画像解析インターフェース

## 1. 背景

- IroMap を単なる色空間可視化サイトではなく、画像から意味のある色解析結果を返すサービスとして定義したい
- 人間向けブラウザUIだけでなく、AI / エージェントが入力と出力を扱える公開API導線が必要
- 可視化だけでは機械的な解釈が不安定なため、構造化されたテキストとJSONを併設する必要がある
- 公開APIとして運用する以上、Vercel上で破綻しないサイズ制限とレート制限が必要

## 2. スコープ

### In Scope

- ブラウザから画像をアップロードして解析できる
- API経由で画像バイナリをリクエストボディへ直接POSTして解析できる
- 解析結果を人間向けHTMLとして構造化して返せる
- 解析結果をAI / エージェント向けJSONとして返せる
- 可視化の意味説明をテキストとして返せる
- 元画像を永続保存せず、その場で解析して破棄する前提を定義する
- 公開APIとして最低限の入力上限とIPベースのレート制限を設ける
- API利用方法を人間とLLMの両方が見つけられる形で公開する

### Out of Scope

- 元画像の永続保存
- 結果の長期保存URL発行
- 画像本体をクエリパラメータへ埋め込む入力方式
- 画像URL取得による解析
- LLMによる自由文診断の自動生成
- 過去バージョンのJSON互換性維持

## 3. 利用者 / ステークホルダー

- ブラウザで画像の色傾向を確認したい一般ユーザー
- IroMap を外部ワークフローへ組み込みたいAI / エージェント利用者
- 解析結果の一貫性と運用コストを担保したい開発者 / 運用者

## 4. ユーザーストーリー

- ユーザーとして、ブラウザから画像をアップロードして結果をページで確認したい。なぜなら可視化と説明を同時に読みたいから
- AIエージェントとして、画像バイナリをプログラム的に送信してJSON結果を受け取りたい。なぜなら後続処理へ接続したいから
- AIエージェントとして、HTML結果からも主要な要約や図の意味を読めるようにしたい。なぜなら描画依存を避けたいから
- 開発者として、画像保存なしでも解析機能を成立させたい。なぜならストレージコストと管理負荷を抑えたいから
- 開発者として、公開APIを無制限に叩かれないようにしたい。なぜならVercel運用でコストと可用性を守りたいから

## 5. 機能要求

### FR-1: 画像入力を機械的に受け付けできる

#### 受け入れ条件

- ブラウザUIから画像ファイルを指定して解析開始できる
- API経由で画像バイナリをリクエストボディへ直接POSTして解析開始できる
- APIは `image/jpeg` `image/png` `image/webp` の `Content-Type` を受け付けできる
- ブラウザUIとAPIが同一画像バイト列を入力とした場合、同一の解析結果モデルを生成できる

### FR-2: 解析結果を構造化HTMLとして返せる

#### 受け入れ条件

- 結果HTMLに見出し、サマリー、数値結果、図の説明を含められる
- 主要色、平均明度、平均彩度、分布傾向などの主要項目をテキストとして明示できる
- canvasや画像が読めない場合でも、HTMLテキストだけで主要な解析内容を把握できる

### FR-3: 解析結果をJSONとして返せる

#### 受け入れ条件

- JSONに入力情報、要約、解釈、可視化用データを含められる
- dominant colors、brightness/saturation系の集計値、分布傾向を機械可読なキーで返せる
- HTMLとJSONで、同じ解析意味を矛盾なく表現できる

### FR-4: 可視化結果の意味説明をテキストで返せる

#### 受け入れ条件

- 各可視化ごとに「何を表示しているか」を説明できる
- 使用色空間、軸、読み方をテキストで明示できる
- 可視化から読み取れる特徴を短い解釈文として返せる

### FR-5: AI / エージェント向けサマリーを返せる

#### 受け入れ条件

- 主要色、画像サイズ、平均明度、平均彩度、特徴文を短い要約ブロックとして返せる
- サマリーはHTMLとJSONの両方に含められる
- サマリーだけ読んでも、画像の大まかな色傾向を把握できる

### FR-6: 元画像を永続保存せずに解析を完結できる

#### 受け入れ条件

- 元画像は解析処理にのみ利用される
- 解析完了後は、元画像を永続ストレージへ保存しない前提で成立する
- 再参照には再アップロードまたは再入力が必要であることを仕様として扱う

### FR-7: 公開APIとして制御可能な入出力インターフェースを提供できる

#### 受け入れ条件

- AI利用者が入力方法、出力形式、制限事項を判別できる導線を提供できる
- JSONレスポンスは現在デプロイ中の実装と一致するトップレベル構造を持つ
- 異常系では、成功時と区別できるエラー情報を返せる

### FR-8: 公開APIの濫用を抑制できる

#### 受け入れ条件

- IP単位でレート制限を適用できる
- 画像サイズと画素数に上限を設けられる
- 制限超過時に `413` または `429` を返せる

## 6. 非機能要求

- NFR-1: 解析処理は単発リクエストで完結し、画像保存を前提としない
- NFR-2: 同一画像入力に対して主要な集計値と要約が再現可能である
- NFR-3: HTML出力は人間可読性を保ちつつ、機械的に見出しや項目を抽出しやすい構造を持つ
- NFR-4: JSON出力は最新デプロイに追従するクライアントが解釈できる構造である
- NFR-5: Vercel Functions の `4.5 MB` request body 上限内で安全に受け付けられる入力サイズを定義する
- NFR-6: 1リクエストあたりの解析時間上限を設定し、タイムアウト前に失敗を返せる

## 7. 業務ルール / 制約

- IroMap の本質は「画像入力から解析結果を返すサービス」として扱う
- 元画像はクエリパラメータに埋め込む主設計にしない
- API入力は画像URLではなく、HTTPボディに格納された画像バイナリのみを扱う
- 出力は見た目だけでなく意味を返す
- HTMLとJSONのどちらか一方にのみ重要な意味情報を閉じ込めない
- 色空間解析はサーバー側で完結可能な構成を優先する
- JSON互換性は過去バージョンまで保証しない
- APIは公開し、初期段階では認証なしで提供する
- レート制限の初期値は `IPごとに 10 req/min` を目安とする
- 画像入力サイズは `4.0 MB` 以下をアプリケーション上限とし、Vercel Functions の `4.5 MB` 制限より低く抑える
- 画像画素数は `12 megapixels` 以下を初期上限とする
- 解析処理は `15 秒` 以内の完了を目標とし、超過時は失敗で打ち切る
- IP偽装や分散送信による制限回避は、現段階では許容する

## 8. エッジケース

- 単色画像でもサマリーと解釈が破綻しない
- ほぼ無彩色画像でも主要色や彩度傾向を返せる
- 非対応フォーマットや壊れた画像入力に対してエラー応答を返せる
- `Content-Type` 詐称の入力に対してデコード失敗または形式エラーを返せる
- `4.0 MB` 超過または画素数超過の入力に対して制限超過エラーを返せる
- レート制限超過時に制限エラーを返せる

## 9. 成功指標

- 人間ユーザーがHTML結果だけで「何が分かったか」を説明できる
- AI / エージェントがJSON結果だけで主要な特徴量を取得し、後続処理へ利用できる
- 同一の解析意味をHTMLとJSONの両方で提示できる
- 保存なし前提でも、画像解析機能として利用価値が成立する
- 公開APIを低コストで継続運用できる
