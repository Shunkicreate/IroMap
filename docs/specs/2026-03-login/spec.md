# Feature Spec: ログイン

## 1. 背景
- 登録済みユーザーが認証されて機能へアクセスする導線が必要
- ユーザー登録機能と合わせて認証フローを成立させる必要がある
- 未整備のままだとアクセス制御の前提が欠ける

## 2. スコープ
### In Scope
- メールアドレスとパスワードでログインできる
- 認証成功時にアクセストークンを発行する

### Out of Scope
- ソーシャルログイン
- 多要素認証

## 3. 利用者 / ステークホルダー
- 登録済みユーザー
- セキュリティ担当
- CS

## 4. ユーザーストーリー
- ユーザーとして、登録済みアカウントでログインしたい。なぜならサービス機能を利用したいから。
- ユーザーとして、未認証メールのままログイン成功したくない。なぜならアカウント保護が必要だから。

## 5. 機能要求
### FR-1
ユーザーはメールアドレスとパスワードでログイン申請できる。

#### 受け入れ条件
- 入力フォーマット不正は 400 を返す
- 認証失敗時は汎用エラーメッセージを返す

### FR-2
システムは認証成功時にトークンを返す。

#### 受け入れ条件
- メール認証済みユーザーのみトークン発行する
- トークンは署名付き JWT を使用する

## 6. 非機能要求
- セキュリティ: account enumeration を避ける
- 性能: ログイン API の p95 応答時間 200ms 以下
- 可観測性: 成功率、失敗率、理由別失敗数を計測する

## 7. 業務ルール / 制約
- `pending` ユーザーはログイン不可
- 連続失敗時の制限は別機能で扱う

## 8. エッジケース
- パスワード誤入力連打
- 期限切れ/不正トークンの利用
- 削除済みアカウントのログイン

## 9. 成功指標
- ログイン成功率
- 認証エラー率
- CS 問い合わせ件数

## 10. 未決事項
- リフレッシュトークン導入の要否
- ログイン試行回数制限のしきい値
